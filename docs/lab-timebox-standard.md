# Lab Timebox Standard

### Purpose

This document defines the time limit for every lab in the Windows and Azure administrator lab series.

The labs must be practical, focused and realistic, but they must not become oversized projects.

### Maximum Duration

Each required lab must be designed to take:

```text
60 to 90 minutes maximum
```

This includes:

* reading the lab brief
* completing the technical tasks
* performing the required verification
* completing one controlled break/fix exercise, where relevant
* collecting safe evidence
* answering the seven reflection questions

The learner should not be expected to spend additional time writing the final report. Final documentation is handled separately by the assistant using the learner's evidence and answers.

### Lab Design Rule

Every lab brief must be scoped as a focused operational task, not a broad project.

A good lab should have:

| Area | Rule |
| --- | --- |
| Scenario | One clear operational scenario |
| New content | One primary skill or closely related skill group |
| Cumulative drill | A limited review of earlier skills, not a full recap of everything |
| Break/fix | One controlled fault unless unsafe, destructive or irrelevant |
| Evidence | Only the evidence needed to prove the outcome |
| Reflection | Exactly seven reflection questions |

### Required Versus Optional Work

Anything likely to push the lab beyond 90 minutes must be marked as optional stretch work.

Required work should be labelled:

```text
Required
```

Optional extra work should be labelled:

```text
Optional stretch task
```

Optional stretch tasks must not be needed to complete the lab.

### Scope Control

When creating a lab, keep the required work inside the timebox by limiting:

* the number of servers or VMs used
* the number of Azure resources created
* the number of scripts required
* the number of break/fix faults introduced
* the number of screenshots requested
* the amount of reading required before starting
* the amount of written explanation expected from the learner

### Split Rule

If a lab cannot realistically be completed within 60 to 90 minutes, split it into multiple labs.

Examples:

| Oversized lab | Split into |
| --- | --- |
| Build AD DS, DNS, users, groups, OUs and GPOs in one session | AD DS and DNS baseline, then users/groups/OUs, then GPO enforcement |
| Configure Intune enrolment, compliance, configuration profiles and remediation scripts | Enrolment baseline, then compliance, then configuration/security baseline, then remediation |
| Deploy Azure VM, networking, monitoring, backup and security controls | Secure VM deployment, then monitoring, then backup, then security hardening |

### Difficulty Standard

The labs should remain technically serious.

The 90-minute limit does not mean the labs should be easy. It means each lab should test a narrow set of skills deeply enough to produce evidence, troubleshooting practice and interview-ready learning.

### Completion Standard

A lab is correctly scoped when the learner can finish the required technical work, gather evidence and answer the seven reflection questions in one focused session of no more than 90 minutes.
